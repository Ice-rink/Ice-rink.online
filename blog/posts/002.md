# 高性能屏蔽词过滤插件的设计思路

## 背景

你有没有遇到过这样的场景：聊天室里总有人发一些不合时宜的词，或者评论区突然冒出几条奇怪的广告？这时候，我们就需要一个“小卫士”来帮我们自动识别并屏蔽这些敏感词。

但问题来了——如果敏感词库里有上万条规则，用户每次发消息都要逐一比对，那服务器岂不是要累哭啦？

所以，今天我们就来聊聊，如何设计一个**既轻巧又高效**的敏感词过滤系统。我会尽量用甜甜的比喻，把算法这颗“硬糖”泡软了喂给你吃～ ✨

## 技术选型

为了让我们的“小卫士”跑得飞快，我们选了三件秘密武器：

- **Trie 树（字典树）**：基础数据结构。你可以把它想象成一棵“字母树”，每个节点代表一个字符。所有共享相同前缀的词语会合并成一条路径，就像树枝分叉一样。这样，查找一个词是否存在时，只需要沿着树干走一遍就行，不用每次都翻遍整个词库。🌳

- **AC 自动机**：多模式匹配的加速器。如果说 Trie 树是“一本字典”，那 AC 自动机就是“一本自带索引的字典”。它能让我们的扫描指针在文本中一路向前，绝不回头。一旦发现匹配失败，它会顺着一条神奇的“失败指针”跳到另一个可能匹配的位置，避免从头再来。这一步让时间复杂度降到 O(n)，其中 n 是待检测文本的长度，处理再多敏感词也几乎不拖慢速度！⚡

- **内存优化：双数组 Trie**：当词库非常大时，Trie 树里的节点和指针会吃掉不少内存。双数组 Trie 用两个简单的数组（base 和 check）把整棵树压缩起来，占用空间更小，查询速度依然很快。这一步属于进阶优化，适合拥有海量敏感词的场景。💾

## 实现细节

我们先从最核心的代码骨架开始，感受一下这个“小卫士”是如何诞生的：

```js
class WordFilter {
  constructor() {
    // 初始化时，我们种下一棵空空的 Trie 树
    this.trie = new Trie();
  }

  // 添加敏感词：像把一颗颗小星星挂上树枝
  addWord(word) {
    let node = this.trie.root;
    for (const char of word) {
      if (!node.children[char]) {
        node.children[char] = new TrieNode();
      }
      node = node.children[char];
    }
    node.isEnd = true; // 标记这是一个完整敏感词的结尾
  }

  // 过滤文本：扫描并替换敏感词，变身小星星！

    // 暂时留个悬念，下一篇再详细展开哦 (ฅ´ω`ฅ)
    return text;
  }
}

class TrieNode {
  constructor() {
    this.children = {};
    this.isEnd = false;
    this.fail = null; // 失败指针，AC 自动机的秘密通道
  }
}
```

看到了吗？代码结构其实不复杂。TrieNode 就像树上的一个小节点，children 里装着它连接的下一层字符，isEnd 表示“这个节点是一个敏感词的终点”。而 fail 字段，就是我们接下来要讲的 AC 自动机里那条神奇的“秘密通道”。

当我们把敏感词一个个 addWord 进去后，就得到了一棵枝繁叶茂的树。再配合 AC 自动机建好失败指针，就能让 filter 方法在文本中轻快地跳跃查找，遇到敏感词就替换成 ***，温柔又果断。
